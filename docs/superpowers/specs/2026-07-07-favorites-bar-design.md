# Дизайн: панель избранных ссылок на new tab

Дата: 2026-07-07
Статус: Утверждено пользователем, ожидает review записанной спецификации

## Контекст

Текущее расширение заменяет страницу новой вкладки и показывает DTF-очередь через
`src/newtab.html`, `src/newtab.js` и `src/newtab.css`. Состояние очереди живет
отдельно в `chrome.storage.local` под ключом `dtfQueueState`; UI не использует
фреймворк, service worker или options page.

Новая функция добавляет верхнюю панель пользовательских избранных ссылок. Это не
браузерные bookmarks, а ссылки, которые пользователь сохраняет внутри расширения.
Добавление должно происходить прямо на new tab через кнопку `+`.

## Цели

1. Показать сверху страницы адаптивную сетку избранных ссылок.
2. Дать пользователю добавлять ссылки через `+` на new tab.
3. Открывать избранную ссылку в текущей вкладке.
4. Поддержать полный редактор: URL, подпись, режим иконки, custom image URL,
   ручной цвет, удаление и порядок.
5. Автоматически получать favicon сайта, показывать букву домена при fallback и
   позволять заменить иконку собственной картинкой.
6. Автоматически подбирать фон плитки по иконке, но разрешать ручную правку цвета.
7. Хранить первую версию локально, но оставить модель пригодной для будущего
   перехода на Chrome Sync.

## Не цели

- Не использовать браузерные bookmarks API.
- Не добавлять отдельную options page в первой версии.
- Не добавлять service worker или фоновые задачи.
- Не делать drag-and-drop в первой версии; порядок меняется кнопками left/right.
- Не синхронизировать данные через `chrome.storage.sync` в первой версии.
- Не реализовывать импорт, экспорт или шаринг набора избранного.

## Архитектура

Favorites должны быть отдельной подсистемой, параллельной DTF-очереди:

| Файл | Ответственность |
| --- | --- |
| `src/favoritesStore.js` | Читает, валидирует и сохраняет состояние избранного. |
| `src/favoritesService.js` | Реализует add/update/delete/move, нормализацию URL и правила иконок/цвета. |
| `src/newtab.js` | Рендерит две зоны: favorites bar сверху и существующую DTF-панель ниже. |
| `src/newtab.css` | Добавляет layout панели, плиток, редактора и адаптивность. |

Состояние очереди не меняется. Сброс DTF-очереди не должен удалять избранное.
Favorites используют свой storage key, например `dtfFavorites`.

`favoritesStore` должен принимать storage area как зависимость, по аналогии с
`createQueueStore(storageArea)`. В первой версии в него передается
`chrome.storage.local`; позже можно будет заменить storage area или добавить
миграцию на `chrome.storage.sync` без переписывания UI.

## UI

Панель расположена сверху страницы, над DTF-карточкой. Если избранных нет, видна
только компактная кнопка `+`. Если элементы есть, показывается сетка плиток и
плитка `+` после них.

Клик по обычной плитке открывает сохраненный URL в текущей вкладке. Это
намеренно отличается от DTF-кнопки `Перейти`, которая открывает статью в новой
вкладке: избранное работает как быстрый переход с new tab.

### Геометрия плиток

Плитка имеет фиксированную комфортную высоту, например `56px`, и скругленные
углы. Ширина адаптивная:

- максимальная ширина: `2 * height`;
- минимальная ширина: `height`;
- пока места хватает, плитки могут быть горизонтальными с отношением сторон 2:1;
- при сужении контейнера плитки пропорционально ужимаются до квадрата;
- новая строка появляется только когда квадратные плитки уже не помещаются.

Иконка внутри плитки не растягивается вместе с плиткой. У нее есть отдельный
максимальный размер, например `28-32px`, и внутренние отступы, чтобы favicon не
заполнял всю кнопку.

Практическая CSS-модель:

```css
.favorites-grid {
  --favorite-tile-height: 56px;
  display: grid;
  grid-template-columns:
    repeat(
      auto-fit,
      minmax(var(--favorite-tile-height), calc(var(--favorite-tile-height) * 2))
    );
}
```

Если реальный CSS grid будет давать нежелательное распределение на крайних
ширинах, допустима маленькая JS-функция, которая выставляет ширину плиток по
размеру контейнера. Предпочтение остается за CSS-only решением, если оно пройдет
визуальную проверку.

### Редактирование

Рядом с панелью есть кнопка-шестеренка. Она открывает режим редактирования или
окно редактирования, где можно:

- изменить URL;
- изменить label;
- выбрать режим иконки: favicon сайта, буква, custom image;
- задать custom image URL;
- изменить цвет фона вручную;
- удалить элемент;
- сдвинуть элемент влево или вправо.

Drag-and-drop остается будущим улучшением. Для первой версии кнопки left/right
проще, доступнее и надежнее в plain DOM UI.

## Модель данных

Состояние хранится отдельно от очереди:

```js
{
  version: 1,
  items: [
    {
      id: "stable-id",
      url: "https://example.com/path",
      label: "Example",
      domain: "example.com",
      iconMode: "favicon",
      customIconUrl: null,
      backgroundColor: "#24292f",
      backgroundColorSource: "auto",
      createdAt: "2026-07-07T00:00:00.000Z",
      updatedAt: "2026-07-07T00:00:00.000Z"
    }
  ]
}
```

Допустимые значения:

- `iconMode`: `"favicon" | "letter" | "custom"`;
- `backgroundColorSource`: `"auto" | "manual"`;
- `customIconUrl`: `string | null`.

Порядок элементов равен порядку массива `items`.

## Нормализация и валидация

Основной URL:

- принимает только `http:` и `https:`;
- если пользователь ввел `example.com`, сохраняется как `https://example.com/`;
- пустые строки и непарсибельные URL отклоняются;
- `domain` извлекается из `URL.hostname`;
- дефолтный `label` берется из домена, но может быть изменен вручную.

Custom image URL:

- принимает только `http:` и `https:`;
- не обязателен, если выбран режим `favicon` или `letter`;
- если картинка не загрузилась, UI показывает fallback и не ломает плитку.

## Favicons

Для Manifest V3 следует использовать официальный Chrome favicon endpoint:

```text
chrome-extension://<EXTENSION_ID>/_favicon/?pageUrl=<ENCODED_PAGE_URL>&size=<SIZE>
```

Для этого в `manifest.json` нужно добавить permission `"favicon"`. Официальная
документация: <https://developer.chrome.com/docs/extensions/how-to/ui/favicons>.

Не использовать старый `chrome://favicon2` как основную реализацию.

## Цвет фона

Фон плитки наследуется из favicon или custom image. Алгоритм локальный:

1. Загрузить изображение.
2. Нарисовать его в маленький canvas.
3. Посчитать заметный dominant/vibrant цвет.
4. Игнорировать прозрачные, почти белые и слишком слабые пиксели.
5. Проверить контраст для буквы или внутреннего элемента.
6. Если чтение пикселей невозможно или картинка не загрузилась, вернуть
   стабильный fallback-цвет от домена.

Ручной цвет имеет приоритет над автоматическим. Если пользователь вручную меняет
цвет, `backgroundColorSource` становится `"manual"` и последующая перезагрузка
favicon не должна перетирать выбранный цвет.

Алгоритм не нужно изобретать как бренд-систему. Достаточно небольшого helper
класса функций: dominant color extraction, contrast check, fallback by domain.
Если потребуется зависимость, ее нужно оценить отдельно; первая попытка может быть
реализована локально из-за малого объема задачи.

## Ошибки

- Нет favicon: показывать первую букву домена.
- Не удалось вычислить цвет: использовать fallback-цвет от домена.
- Custom image не загрузился: показать fallback и дать сменить режим/URL.
- Ошибка сохранения в storage: показать ошибку в редакторе, не менять уже
  сохраненное состояние.
- Битое persisted-состояние: сбросить только favorites-состояние до пустого,
  не трогая DTF-очередь.

## Тестирование

Обязательные unit-тесты:

- `favoritesStore.test.js`: валидное состояние, битое состояние, unknown fields,
  лимиты, сохранение и чтение.
- `favoritesService.test.js`: add, update, delete, move left/right, URL
  normalization, icon mode rules, manual color priority.
- `manifest.test.js`: наличие permission `"favicon"` после реализации.

Желательные проверки UI:

- панель пустого состояния показывает `+`;
- сохраненные элементы рендерятся до DTF-панели;
- клик по плитке меняет `window.location.href` на сохраненный URL;
- edit mode показывает действия редактирования без конфликта с открытием ссылки;
- сетка проходит визуальную проверку на широком и узком viewport.

Если текущий Node test setup не поддержит DOM-тесты без тяжелого browser harness,
достаточно unit-тестов store/service, `npm run check`, `npm test` и ручной
load-unpacked проверки в Chromium.

## Рекомендуемый порядок реализации

1. `favoritesStore` и тесты валидации persisted-состояния.
2. `favoritesService` и тесты CRUD/нормализации.
3. Минимальный UI: пустая панель, `+`, add URL, render плиток, открытие в текущей
   вкладке.
4. Режим редактирования: update/delete/move, label и custom image URL.
5. Favicon endpoint и permission `"favicon"`.
6. Auto color helper, fallback-цвет и manual color override.
7. CSS-полировка адаптивной геометрии плиток.
8. Документация README/privacy при необходимости, особенно если изменится список
   permissions.

## Оценка сложности

Сложность средняя для этой кодовой базы. Хранение и CRUD простые, потому что уже
есть паттерн `chrome.storage.local` и unit-тесты. Основной риск находится в UI:
адаптивная геометрия плиток, редактор без конфликтов с открытием ссылки и
автоматический цвет из favicon.

Разумно делать фичу итеративно: сначала CRUD + сетка + favicon/letter fallback,
затем custom image, ручной цвет и auto color extraction.
