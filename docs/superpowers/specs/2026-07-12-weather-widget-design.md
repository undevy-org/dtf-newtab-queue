# Дизайн: погодная секция под новостной карточкой

Дата: 2026-07-12
Статус: Утверждено пользователем, ожидает review записанной спецификации

## Контекст

Расширение заменяет страницу новой вкладки (`src/newtab.html`, `src/newtab.js`,
`src/newtab.css`) и уже показывает две независимые подсистемы на одной
странице: панель избранных ссылок сверху (`favoritesStore.js` /
`favoritesService.js`, `chrome.storage.sync`) и DTF-очередь новостей
(`queueStore.js` / `queueService.js` / `dtfApi.js`, `chrome.storage.local`).
`host_permissions` сейчас ограничены `https://api.dtf.ru/*`.

Новая функция добавляет третью независимую секцию — погоду — визуально под
DTF-карточкой (`#app`). Репозиторий, в котором реализуется фича — именно
`dtf-newtab-queue` (не параллельная platform-версия `~/newtab-widgets`): это
осознанный выбор пользователя, сделанный при брейнсторминге, несмотря на то,
что platform-репо существует именно для будущих виджетов такого рода.

## Цели

1. Показать под новостной карточкой отдельную панель погоды с четырьмя
   метриками: температура, UV-индекс, вероятность дождя сегодня, качество
   воздуха.
2. Получать все данные из бесплатного API без ключа — Open-Meteo (weather +
   air quality + geocoding), чтобы не хранить секрет в публичном клиентском
   коде.
3. Пользователь один раз указывает город вручную (без permission-промпта
   геолокации браузера); выбор города переживает переустановку вкладки и
   синхронизируется между устройствами пользователя.
4. Кэшировать ответ на 30 минут, чтобы не дёргать API при каждом открытии
   вкладки.
5. Изолировать отказ погодной секции от новостной карточки: сбой Open-Meteo
   не должен ломать или прятать DTF-виджet.

## Не цели

- Не использовать `navigator.geolocation` и не запрашивать permission на
  геолокацию браузера.
- Не использовать IP-based геолокацию (потребовала бы ещё один внешний сервис
  и не даёт точности выше города).
- Не использовать IQAir/AirVisual или OpenWeatherMap — оба требуют API-ключ,
  который был бы виден в публичном репозитории.
- Не делать краткосрочный nowcast дождя (`"дождь через 20 минут"`). Только
  дневная агрегированная вероятность.
- Не поддерживать несколько городов одновременно — один активный город на
  пользователя.
- Не добавлять фоновое обновление через `alarms`/service worker. Обновление
  происходит только при открытии new tab и только если кэш устарел.
- Не переключать единицы измерения (°C/°F, км/ч и т.д.) — фиксированный
  метрический формат, аудитория русскоязычная.
- Не показывать почасовой прогноз или многодневный форекаст — только текущее
  значение + агрегат на сегодня.

## Архитектура

Погода — отдельная подсистема, параллельная favorites и очереди, по тому же
паттерну `api` / `store` / `service` (+ `uiState` для состояния формы ввода
города):

| Файл | Ответственность |
| --- | --- |
| `src/weatherApi.js` | Чистые функции `fetchWeather()`, `fetchAirQuality()`, `geocodeCity()`. Валидация и нормализация ответа, `WeatherApiError` (по образцу `DtfApiError`). Принимает `fetchImpl` как зависимость. |
| `src/weatherStore.js` | Читает/пишет город (`chrome.storage.sync`) и кэш последнего ответа (`chrome.storage.local`) раздельно. |
| `src/weatherService.js` | Оркестрация: TTL-проверка кэша, геокодинг при смене города, параллельный `Promise.all` для weather+air-quality, сборка единого состояния для рендера. |
| `src/weatherUiState.js` | Состояние формы ввода/смены города: `idle` / `editing`, по образцу `favoritesUiState.js`. |
| `src/newtab.js` | Рендерит третью секцию `#weather` под `#app`; обрабатывает submit формы города и клик по кнопке смены города. |
| `src/newtab.css` | Стили `.weather-panel`, `.weather-metrics`, `.weather-metric`. |

Ни `queueStore`/`queueService`, ни `favoritesStore`/`favoritesService` не
меняются. Сбой инициализации погоды не должен трогать состояние DTF-очереди
или избранного.

## Данные Open-Meteo

Все три эндпоинта не требуют ключа для non-commercial use (лимит свободного
тарифа — 10 000 запросов/день, 600/мин; для одного пользователя на новой
вкладке это на порядки больше реальной нагрузки).

### Погода + UV + дождь

```
GET https://api.open-meteo.com/v1/forecast
  ?latitude=<lat>&longitude=<lon>
  &current=temperature_2m
  &daily=uv_index_max,precipitation_probability_max
  &timezone=auto
```

Используемые поля ответа: `current.temperature_2m`, `daily.uv_index_max[0]`,
`daily.precipitation_probability_max[0]`. `weather_code` (иконка состояния —
солнце/облачно/дождь) сознательно не запрашивается: в целях этой итерации нет
condition-иконки, только четыре перечисленные метрики. Добавить его —
тривиальное расширение `weatherApi.js`, если понадобится позже.

### Качество воздуха

```
GET https://air-quality-api.open-meteo.com/v1/air-quality
  ?latitude=<lat>&longitude=<lon>
  &current=european_aqi,pm2_5
```

Используемые поля: `current.european_aqi`, `current.pm2_5`.

Категории European AQI (жёстко зашитые пороги в `weatherApi.js`, так как API
их не возвращает текстом):

| Индекс | Категория (label) |
| --- | --- |
| 0–20 | Хорошо |
| 20–40 | Приемлемо |
| 40–60 | Умеренно |
| 60–80 | Плохо |
| 80–100 | Очень плохо |
| >100 | Критично |

### Геокодинг города

```
GET https://geocoding-api.open-meteo.com/v1/search
  ?name=<город>&count=1&language=ru
```

Берётся первый результат (`results[0]`): `latitude`, `longitude`, `name`,
`country`. Пустой `results` (город не найден) — ошибка `WeatherApiError` с
понятным сообщением, форма остаётся открытой для повторного ввода.
Геокодинг вызывается **только** при первичной настройке города и при явной
смене города пользователем — не при каждом обновлении погоды.

## Модель данных

Город — `chrome.storage.sync`, ключ `dtfWeatherLocation`:

```js
{
  version: 1,
  name: "Тбилиси",
  country: "Georgia",
  latitude: 41.72,
  longitude: 44.78
}
```

Кэш последнего ответа — `chrome.storage.local`, ключ `dtfWeatherCache`:

```js
{
  version: 1,
  locationName: "Тбилиси",   // для инвалидации при смене города
  fetchedAt: 1783900000000,   // epoch ms
  temperature: 24,
  uvIndexMax: 6.1,
  precipitationProbabilityMax: 20,
  europeanAqi: 34,
  pm2_5: 11.4
}
```

Разделение sync/local намеренное: город — маленькая долгоживущая настройка,
которую стоит синхронизировать между устройствами (как избранное); кэш —
эфемерные данные, обновляемые каждые 30 минут, которым в `sync`-квоте (мала,
~100 КБ на всё расширение) не место, и синхронизация которых между
устройствами создавала бы гонки без пользы.

## UI

Секция `<section class="panel weather-panel" id="weather" aria-live="polite">`
рендерится в `newtab.html` сразу после `#app`, использует уже существующие
классы `.panel`/`.title`/`.meta`/`.status` для визуальной консистентности с
новостной карточкой.

### Пустое состояние (город не задан)

Заголовок «Укажите город» + инлайн-форма с одним текстовым полем
(`.favorite-input`) и кнопкой submit (`.button.button--primary`). До ввода
города сетевых запросов к Open-Meteo нет вообще.

### Заполненное состояние

Заголовок — название города. Строка из четырёх компактных метрик
(`.weather-metrics` → `.weather-metric` × 4):

1. Температура, `24°`
2. UV-индекс со словесным уровнем (низкий/умеренный/высокий/оч. высокий/
   экстремальный — стандартная шкала ВОЗ: 0–2/3–5/6–7/8–10/11+)
3. Вероятность дождя сегодня, `20%`
4. AQI со словесной категорией из таблицы выше, плюс `PM2.5` мелким текстом
   под категорией (единственное сырое число качества воздуха, которое
   показывается пользователю — ближе к тому, как это делает IQAir)

Рядом с заголовком — иконка-карандаш (`createIconNode("pencil")`, как в
`favorites-panel__row`) для смены города; клик открывает ту же форму, что и
пустое состояние, предзаполненную текущим названием.

## Кэширование и обновление

При инициализации `weatherService`:

1. Если город не задан в `weatherStore` — рендер пустого состояния, запросов
   нет.
2. Если задан — читается `dtfWeatherCache`. Если `locationName` совпадает с
   текущим городом и `Date.now() - fetchedAt < 30 * 60 * 1000` — рендер из
   кэша немедленно, без сетевого ожидания.
3. Если кэша нет, кэш устарел или город сменился — параллельный
   `Promise.all([fetchWeather(...), fetchAirQuality(...)])`, результат
   нормализуется, пишется в `dtfWeatherCache` с новым `fetchedAt`, рендерится.

Смена города пользователем всегда форсирует шаг 3 (геокодинг + немедленный
фетч), независимо от TTL.

## Ошибки

- Город не найден при геокодинге — `WeatherApiError`, форма города остаётся
  открытой, статус `createStatus(text, { error: true })` внутри формы.
- Сеть недоступна / Open-Meteo вернул не-200 / невалидный JSON:
  - если есть валидный кэш для текущего города — рендерится кэш с
    ненавязчивой пометкой «Не удалось обновить» вместо падения в пустоту
    (stale-while-error);
  - если кэша нет — секция показывает `createStatus(text, { error: true })`
    с текстом ошибки вместо метрик.
- Битый/невалидный persisted-кэш или город (не прошли валидацию в
  `weatherStore`) — сбрасывается только погодное состояние, DTF-очередь и
  избранное не трогаются (та же изоляция, что уже принята для favorites).
- Ошибка инициализации погоды никогда не блокирует рендер `#app` (новостной
  карточки) и `#favorites` — независимые try/catch вокруг каждой подсистемы
  в `newtab.js`, как это уже сделано для favorites сегодня.

## Манифест

`manifest.json` → `host_permissions` пополняется тремя явными хостами (тот же
принцип минимальной area, что и у `api.dtf.ru`, без wildcard-домена):

```json
"host_permissions": [
  "https://api.dtf.ru/*",
  "https://api.open-meteo.com/*",
  "https://air-quality-api.open-meteo.com/*",
  "https://geocoding-api.open-meteo.com/*"
]
```

Это расширение permissions — существующие пользователи при обновлении увидят
в Chrome стандартный prompt о новых возможностях расширения. Не блокер для
реализации, но стоит упомянуть в CHANGELOG при релизе.

Новый `permissions` (`storage`) не требуется — оба storage-API уже
задействованы существующими подсистемами.

## Тестирование

Обязательные unit-тесты (`node --test`, тот же паттерн, что у
`dtfApi.test.js` / `queueStore.test.js` / `favoritesService.test.js`):

- `weatherApi.test.js`: построение URL и параметров для всех трёх
  эндпоинтов; нормализация валидного ответа; ошибки на не-200, невалидный
  JSON, отсутствующие поля, пустой `results` геокодинга; маппинг
  AQI-индекса на категорию по границам таблицы.
- `weatherStore.test.js`: чтение/запись города и кэша через фейковую
  storage area; раздельная работа sync/local; невалидное сохранённое
  состояние → сброс только погодной части.
- `weatherService.test.js`: свежий кэш → рендер без сети; устаревший кэш →
  рефетч; смена города → форс-рефетч независимо от TTL; ошибка сети при
  наличии кэша → stale-with-error; ошибка при отсутствии кэша → error-state.
- `manifest.test.js`: наличие трёх новых `host_permissions`.

Желательно, но не обязательно для первой итерации: ручная проверка через
`chrome://extensions` load-unpacked — ввод города, отображение метрик,
поведение при недоступном городе, при офлайне.

## Рекомендуемый порядок реализации

1. `weatherApi.js` + тесты: построение запросов, нормализация, ошибки,
   AQI-категории — без зависимости от Chrome API, легко покрывается тестами
   первым.
2. `weatherStore.js` + тесты: раздельные sync/local ключи, валидация.
3. `weatherService.js` + тесты: TTL-логика, оркестрация фетчей, форс-рефетч
   при смене города.
4. `manifest.json`: три новых `host_permissions` + `manifest.test.js`.
5. Минимальный UI в `newtab.js`/`newtab.html`: пустое состояние, форма
   города, рендер метрик после первого успешного фетча.
6. `weatherUiState.js` + кнопка смены города поверх уже заполненного
   состояния.
7. `newtab.css`: `.weather-panel`, `.weather-metrics`, адаптивность в узком
   viewport.
8. Ручная проверка в браузере + CHANGELOG-запись про новый permission.

## Оценка сложности

Средняя. Сетевой и storage-слой (`weatherApi`/`weatherStore`/`weatherService`)
прямолинейны и почти дословно повторяют уже существующие паттерны
(`dtfApi`/`queueStore`/`queueService`), поэтому основной объём — не
изобретение нового, а копирование проверенной структуры на новый домен
данных. Единственный чуть менее тривиальный кусок — TTL/кэш-инвалидация при
смене города (шаг 3 в разделе «Кэширование и обновление»), но и она
локальная и полностью покрывается unit-тестами без DOM.
